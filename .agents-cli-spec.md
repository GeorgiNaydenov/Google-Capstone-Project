# Clinical AI Command Center Agent Spec

## Overview

Build a full-stack clinician-facing application in which the existing Google
ADK multi-agent system is the workflow engine. The product must support a
synthetic, no-login public demo and make agent routing, tools, evidence,
confidence, human review, storage state, and audit history visible.

The existing `capstone_agent` package remains the backend foundation. ADK Web
is a developer surface, not the product UI.

## Example Use Cases

1. Upload a synthetic session image, observe extraction agents, edit or approve
   fields, and see JSON, relational, vector, timeline, and audit state update.
2. Ask a patient-scoped multimodal question and receive a confidence-scored
   answer with reopenable text and image citations.
3. Ask a population question, preview generated read-only SQL and its safety
   result, execute it, and receive a table, chart, CSV export, history, and audit.
4. Switch between Clinician and Admin roles without logging out while
   preserving the relevant patient or session context.

## Architecture

- Frontend: React/Vite clinical application with 16 routed screens.
- Product API: FastAPI routes and OpenAPI contracts.
- Agent engine: existing ADK App, root orchestrator, and three pipelines.
- State: one mutable deterministic demo repository per demo session.
- Events: persisted agent-run and step events delivered by SSE with polling
  fallback.
- Production providers: ports for GCS, Firestore/JSON, Cloud SQL or BigQuery,
  Vertex AI Vector Search, Gemini vision, and Cloud Logging.
- Primary deployment: one Cloud Run artifact serving compiled UI and API.
- Optional interoperability: existing A2A server and MCP tools.

## Tools Required

- image upload, metadata, quality, OCR, and multimodal analysis
- clinical field structuring and deterministic validation
- review request, edit, approve, reject, and re-run transitions
- JSON, relational, vector, asset, timeline, and audit persistence
- patient/session lookup and scoped text/image/vector retrieval
- citation creation and authorized evidence access
- schema discovery, SQL generation, deterministic SQL safety, execution
- chart specification, CSV export, query history, and audit

All tool boundaries use Pydantic validation and structured response envelopes.

## Constraints and Safety Rules

- Synthetic data only in public demo mode; never claim PHI readiness.
- Never expose secrets, hidden chain-of-thought, or unauthorized patient data.
- Enforce actor, active role, organization, patient, and session scope before
  every protected API or tool action.
- Treat uploads, OCR, notes, and retrieved evidence as untrusted input.
- LLMs may interpret and synthesize, but deterministic code owns policy,
  authorization, SQL safety, review state, and persistence transitions.
- Clinical record changes require an identified human review decision.
- Writes are idempotent, versioned, auditable, and safe to resume.
- Cross-session ADK memory is not the source of truth for clinical records.
- Operational logs and immutable audit events are separate.
- Deployment requires explicit human approval.

## Success Criteria

- All 16 brief-defined screens are routed, connected, and use shared API data.
- Public demo, role switching, patient navigation, and reset work without login.
- The three guided workflows pass end-to-end browser tests.
- Agent steps, evidence, confidence, review, sync, timestamps, and audit are
  visible and backed by persisted records.
- No primary control is inert or timer-simulated.
- API, deterministic pytest, ADK eval, frontend, accessibility, browser,
  container, and secret-scan gates pass.
- README, architecture, setup, deployment, screenshots, and demo walkthrough
  match current behavior.
